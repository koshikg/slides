What is Managed DevOps Pools?

Overview

Managed DevOps Pools is a fully managed service where VMs or containers powering the agents live in a Microsoft Azure subscription (not your own subscription).

Key Capabilities

  • Custom Agent Images: Use VM images tailored for your CI/CD workloads
  • Geographic Distribution: Deploy agents closest to your dependencies
  • Auto-scaling: Scale up and down based on configuration
  • State Persistence: Maintain agent state up to 7 days for faster builds via cache hits
  • Long-running Workflows: Support workflows up to 2 days long
  • Network Integration: Access company networks or isolate to specific endpoints
  • Additional Storage: Attach data disks without requiring larger SKUs

üéØ Evolution of Azure DevOps VMSS Agents

Managed DevOps Pools simplifies custom pool creation by improving scalability and reliability compared to Azure Virtual Machine Scale Set agent pools.

Pipeline Demands Overview

Demands provide a way for pipelines to specify what capabilities must be present in an agent for Azure DevOps to send a job to that agent. Some demands configure agent attributes, while others route jobs to specific agents.

üéØ Complete Demands Reference

üóÇÔ∏è WorkFolder

Override default agent work folder location (D:\ for Windows, /mnt for Linux)

‚ö° Priority

Set job execution priority: High, Medium (default), or Low

üñºÔ∏è ImageOverride

Specify which image to use when your pool has multiple images

üîÑ ImageVersionOverride

Use a specific version of an image instead of the configured version

üéØ CustomCapabilities

Target specific agent instances in stateful pools for optimized workflows

üìã Basic Demand Examples

pool: name: my-managed-pool demands: - WorkFolder -equals f:\custom-work-folder - Priority -equals High - ImageOverride -equals ubuntu-24.04-gen2

Key Points:

  • Multiple demands can be combined in a single pipeline
  • Don't use quotes around alias names in demands
  • Invalid WorkFolder paths fall back to default location

Detailed Demands Configuration

ÔøΩÔ∏è WorkFolder - Advanced Usage

Configure custom agent working directory, especially useful with attached data disks

pool: name: my-managed-pool demands: - WorkFolder -equals f:\custom-work-folder

Important: Directory is created if not present. Invalid paths fall back to default location.

Use cases: Attached data disks, specific drive requirements, custom folder structures

‚ö° Priority - Job Execution Order

Control job execution priority: High, Medium (default), Low

pool: name: my-managed-pool demands: - Priority -equals High

Behavior:

  • Higher priority jobs execute first in queue
  • Sequential jobs in same pipeline ignore priority
  • Parallel jobs may execute out of priority order if queued simultaneously
  • Perfect for hotfixes and critical deployments

üñºÔ∏è ImageOverride - Multi-Image Pools

Specify which image to use when your pool has multiple images configured

pool: name: my-managed-pool demands: - ImageOverride -equals ubuntu-24.04-gen2

‚ö†Ô∏è Important

Without ImageOverride, pipelines use the first listed image in your pool. Don't use quotes around alias names.

8 / 13

Advanced Pipeline Demands

üîÑ ImageVersionOverride

Use a specific version of an image instead of the configured version

pool: name: my-managed-pool demands: - ImageVersionOverride -equals 20250427.1.0

Use cases:

  • Test new image versions before promoting them to latest
  • Validate pipeline compatibility with specific image versions
  • Rollback to previous known-good image versions
  • Troubleshoot issues potentially caused by image updates

‚ö†Ô∏è Important

Standby agents use configured image versions, so ImageVersionOverride will provision fresh agents on demand, affecting startup time.

üéØ CustomCapabilities (Stateful Pools Only)

Target specific agent instances in stateful pools for optimized workflows

pool: name: my-managed-pool demands: - ImageOverride -equals windows-2022 - CustomCapabilities -equals MyCustomValue

How it works:

  • First run: Tags available agent with CustomCapabilities value
  • Subsequent runs: Routes to same tagged agent if online
  • Perfect for large repository clones and cached dependencies
  • Significantly improves build times for related workflows

üìù Stateful Pools Only

CustomCapabilities only applies to stateful pools. Stateless pools always provide fresh agents.

Cost Management & Performance Optimization

üí∞ Key Cost Optimization Strategies

üñ•Ô∏è Right-size VM Selection

Choose appropriate VM sizes based on actual workload requirements, not maximum potential needs

‚ö° AMD vs Intel Architecture

AMD-powered VMs can be up to 40% less expensive than Intel with same/better performance

üìç Regional Co-location

Deploy pools in same region as dependencies to reduce latency and network costs

üíæ Smart Storage Strategy

Use data disks instead of expensive OS disk upgrades or larger VM SKUs

‚öôÔ∏è VM Selection Guidelines

üîç VM Series Selection

  • Compute Optimized: CPU-intensive workloads (e.g., build compilation)
  • Memory Optimized: Memory-heavy applications (e.g., large datasets)
  • General Purpose: Balanced CPU/memory for most CI/CD workloads
  • Latest Generation: Prefer V5 over V4 for same price with better performance

üìä Standby Agent Cost Management

üí° Optimization Strategies

Option 1: Reduce/Eliminate Standby Agents - Review historical utilization - Turn off for non-time-critical workloads - Accept 5-10 minute agent startup time

Option 2: Stateful Agents with Grace Period - Agents stay online briefly after job completion - Ideal for burst workloads - Balance between performance and cost

Cost Impact: Standby agents run 24/7 - ensure they provide value for your workflow patterns
